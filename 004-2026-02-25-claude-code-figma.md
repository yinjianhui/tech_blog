# 004-2026-02-25 Figma放大招了!用Claude Code写的代码,一键变设计稿

## 引言

想象这样一个场景:

你是一名产品经理,有个想法急需验证。你打开Claude Code,用自然语言快速生成了一个可交互的原型——可能是3种不同的导航方案,或者一个复杂的多步骤表单。代码在本地跑起来了,界面能点能跳,数据也是真实的。

但接下来呢?

如果要给老板演示,你得录屏或者截图。如果要和设计师讨论,你得解释"这个按钮点后会跳到哪"。如果要和开发对齐,你得让他们把代码拉到本地跑一遍。

**这就是"代码优先"工作流的尴尬时刻:快速原型做到了,但协作还没开始。**

就在几天前,Figma发布了一个新功能:**Claude Code to Figma**——把你用Claude Code写出来的代码界面,一键转换成Figma里可编辑的设计稿。

这不是简单的"截图导入",而是标志着**代码与设计的边界正在消融**。更重要的是,它揭示了一个更深层的趋势:**当AI让"做出来"变简单,"做得好"成了新的战场。**

---

## 一、当AI让"做出来"变简单,"做得好"成新战场

过去的几年里,AI正在重新定义"创作"的起点。

还记得吗?以前要做一个原型,你得:
- 画线框图(Axure/Sketch)
- 或者直接写代码(HTML/CSS/React)
- 每一步都不快,每一步都有门槛

现在呢?

**Figma Make**:你用自然语言描述一个界面,AI生成可交互的原型
**Claude Code**:你用自然描述你的需求,AI生成真实的前端代码

创作的门槛被压得极低。**从想法到可触摸的东西,中间的摩擦力几乎消失了。**

但Figma在官方博客里提出了一个很妙的观察:

> "With AI changing what 'starting' looks like, creating has become easier—there's often a concrete prototype to react to almost immediately. This shifts the design conversation from **how to make something** to **which version to expand on**."

翻译过来就是:**AI改变的,不只是"怎么做",而是"做哪个"。**

当"做出来"不再是瓶颈,问题就变成了:
- 你有3个版本的导航栏,哪个更好?
- 你有2种信息架构的方案,哪个更符合用户心智?
- 你有5种表单交互的方式,哪个转化率会更高?

这些问题,靠"代码运行起来"是回答不了的。它们需要**全局视角、对比分析、团队协作、用户反馈**——这些是"画布"的强项,不是"代码"的强项。

所以Figma的思路是:**无论你的起点是Figma Make(从Prompt到原型),还是Claude Code(从Prompt到代码再到界面),终点都应该是"可编辑的设计稿,用于深入探索"。**

这不是技术工具的简单叠加,而是**对AI时代创作流程的重新思考。**

---

## 二、代码与画布的"双剑合璧":收敛与发散的艺术

Figma官方博客里有一段话,我觉得是整篇文章最有洞察的部分:

> "Code is powerful for **converging**—running a build, clicking a path, and arriving at one state at a time. The canvas is powerful for **diverging**—laying out the full experience, seeing the branches, and shaping direction collectively."

翻译一下:
- **代码擅长"收敛"**:运行构建,点击路径,一次到达一个状态
- **画布擅长"发散"**:展开完整体验,看到所有分支,集体塑造方向

这不仅仅是工具特性的差异,而是**两种思维模式的互补**。

### 代码的"收敛式"优势

当你用Claude Code写原型时,你处于一个"收敛模式":
- 你想验证"这个交互行不行"
- 你快速改代码、刷新浏览器、看结果
- **单人快**——所有上下文都在你脑子里
- **线性探索**——一次只能看一个状态

这在早期探索非常高效,但问题也很明显:
- **界面一多,全局视角就丢了**——你可能改了10个版本,但很难把它们放在一起对比
- **协作成本高**——要给别人看,得录屏/截图/让他们本地跑
- **难以"发散"**——想尝试"如果把这个步骤和那个步骤对调",得改半天代码

### 画布的"发散式"价值

当这些界面被导入Figma后,你进入了"发散模式":
- **全局视角**——所有版本并排看,一眼看出模式、gap、不一致
- **可编辑**——不是截图,是可以继续改的Frame
- **可协作**——设计师可以标注,开发可以评论,PM可以画箭头

更重要的是:**你可以"无代码地"探索变体。**

比如你想测试"如果把这个3步流程改成2步会怎样":
- 在代码里:得重构逻辑、改路由、调样式
- 在Figma里:复制Frame,删掉一个,连线改一下,5分钟搞定

**这不是鼓励"不写代码",而是鼓励"在正确的时机写代码"。** 先在画布上发散探索,确定方向后,再收敛到代码实现。

## 三、一个产品经理的真实工作流:从快速原型到深度探索

让我们用一个具体的场景,看看这个新工作流如何改变产品经理的日常工作。

### 场景:你需要验证3种不同的导航方案

假设你负责一个SaaS产品,当前导航是"顶部横栏式",但你怀疑"侧边栏式"可能更适合复杂功能,还想试试"混合式"(顶部一级+侧边二级)。

#### 传统工作流

**方案A:设计先行**
- 和设计师对齐需求 → 设计出3稿 → 评审 → 修改 → 定稿 → 给开发
- **问题**:前期沟通成本高,一旦定稿不好改

**方案B:代码先行**
- 你自己写3个版本 → 截图/录屏给团队看 → 收集反馈 → 自己改代码 → 再演示
- **问题**:团队无法直接参与,你成了瓶颈

#### 新工作流:Claude Code + Figma

**第一步:快速原型(收敛模式)**
你打开Claude Code:
```
"帮我写一个React组件,实现3种导航方式:顶部横栏、侧边栏、混合式。用真实的菜单数据,至少2级导航。"
```

10分钟后,你有了一个本地运行的页面,3种方案都能点能跳。你在浏览器里快速验证:
- 布局是否合理?
- 交互是否顺畅?
- 响应式如何?

**第二步:导入Figma(模式切换)**
你使用"Claude Code to Figma"功能,依次捕获3种方案的界面,复制到剪贴板,粘贴到Figma文件中。

现在你有了:
- 3个可编辑的Frame,不是截图
- 保留了布局、文字、颜色等所有设计元素
- 团队成员可以直接在上面标注、修改

**第三步:全局探索(发散模式)**
你和设计师、开发Lead开了一个Figma会议:
- 把3个方案并排放,一眼看出"侧边栏能容纳更多菜单""混合式学习成本高"
- 设计师标注:"顶部横栏在小屏下太挤,建议加折叠"
- 开发说:"侧边栏的动画效果比较复杂,但如果确定要做,2天能搞定"
- 你复制了"侧边栏"的Frame,改了2个细节:一个是"固定宽度",一个是"可折叠宽度",放在一起对比

**第四步:决策与返工(再次收敛)**
会议结论:采用"可折叠侧边栏",但需要调整图标风格和间距。

你回到Claude Code:
```
"基于刚才的讨论,把侧边栏方案优化:折叠按钮放在顶部,默认宽度240px,折叠后64px,图标用线性风格。"
```

20分钟后,新的版本出来了。你再次导入Figma,团队确认后,交给开发正式实现。

### 为什么这个工作流更高效?

**1. 速度不会因为协作而牺牲**
- 前期用Claude Code快速试错,不需要等设计师排期
- 中期用Figma协作,不需要等所有人都把代码跑起来

**2. 质量不会因为速度而妥协**
- 在Figma里的"发散探索",避免了"代码写了一半才发现方向错了"
- 团队能在早期参与,而不是等成品出来才给反馈

**3. 上下文不会因为切换而丢失**
- 从代码到画布,设计元素完整保留
- 从画布回到代码,决策已经被清晰地记录下来

## 四、不只是"导入导出":双向通道的想象力

如果"Claude Code → Figma"只是单向的,那它只是一个便利工具。但Figma同时发布了**Figma MCP Server**,让这个工作流变成了双向的:

### 正向:Code → Canvas (快速验证)
你用Claude Code写界面 → 导入Figma → 团队协作探索

### 反向:Canvas → Code (设计感知)
你在Figma里设计 → 用MCP Server + AI提示 → 生成"设计感知的代码"

> "Bring work back into a coding environment using a prompt and a link to a Figma frame."

这意味着什么?

**场景1:在Figma里讨论,在代码里实现**
设计师在Figma里调整了一个模块的间距和对齐方式,你不需要再"目测像素"去写CSS,而是:
```
"基于这个Figma链接[链接],生成对应的Tailwind CSS代码,保持设计规范。"
```

**场景2:迭代在画布,实现在代码**
你用Claude Code生成了第一版 → 导入Figma → 团队标注修改意见 → 你在Figma里调整到位 → 用MCP Server生成最终代码

**场景3:设计系统的持续对齐**
设计团队在Figma里维护Design System → 开发用MCP Server实时生成对应的组件库 → 两边始终同步

这不是"取代设计师"或"取代开发者",而是**让设计意图和代码实现之间,有了一个更顺畅的翻译层。**

## 五、这对产品经理和设计师意味着什么?

如果你是产品经理或设计师,这个新功能带来的不只是"方便",而是**工作模式的转变**。

### 对产品经理:从"文档驱动"到"原型驱动"

传统PM工作流:
- 写PRD → 设计出稿 → 开发评审 → 排期实现
- **问题**:前期想得再周全,也难免有遗漏

新工作流:
- 用Claude Code快速搭3个方案 → 导入Figma和团队讨论 → 确定方向 → 写PRD(基于共识)
- **优势**:PRD不再是"拍脑袋",而是"验证后的共识"

### 对设计师:从"执行者"到"探索者"

传统设计流程:
- PM给需求 → 设计出稿 → PM评审 → 修改 → 再评审
- **问题**:设计师常常在"执行别人想法",而非"探索可能性"

新流程:
- PM/开发用Claude Code生成初版 → 导入Figma → 设计师在此基础上探索更好的交互/视觉方案
- **优势**:设计师的创造力用在"提升体验",而非"从零开始"

### 对团队:从"接力棒"到"共创"

传统协作:
- PM → 设计 → 开发,线性传递
- 每个环节之间,信息会丢失、理解会偏差

新协作:
- 在Figma这个"共享画布"上,三方同时面对一个可编辑的稿子
- PM标注用户路径,设计师调整交互细节,开发标注技术风险
- **信息不丢失,理解可对齐**

## 六、但也有需要警惕的地方

新工具很强大,但不是银弹。我看到了几个潜在问题:

### 1. "快速原型"变成"快速产生垃圾"

当"做出来"太容易,可能会:
- 为了求快,跳过必要的用户研究
- 生成了10个版本,但没时间深入思考任何一个
- 团队陷入"选择困难症"

**应对**:明确每个阶段的目标——早期用快速原型验证假设,后期用设计稿打磨细节。

### 2. 设计债务的累积

Claude Code生成的代码,设计质量可能不稳定:
- 间距可能不规范
- 色彩可能不符合品牌
- 组件可能没有复用

**应对**:导入Figma后,让设计师"规范化"一遍,而不是直接用。

### 3. 流程管理的复杂度

不是所有场景都适合这个流程:
- 简单的CRUD页面,可能直接用Design System更快
- 大型重构项目,可能需要完整的信息架构设计
- 高度规范化的B端产品,可能需要严格的组件库管理

**应对**:团队需要明确"什么时候用快速原型,什么时候走标准流程"。

## 七、结语:快速原型+深度迭代=高效创作

Figma的这次更新,表面看是"导入代码界面"的功能,但实质上是**对AI时代创作流程的重新定义。**

过去,创作是线性的:
- 想法 → 原型 → 设计 → 开发
- 每个环节有门槛,每个环节有摩擦

现在,创作是循环的:
- 快速原型(收敛) → 深度探索(发散) → 验证假设(收敛) → 打磨细节(发散)
- AI降低了"原型"的门槛,Figma降低了"探索"的门槛

**当"做出来"不再是难题,"做得好"就成了竞争力。**

对你来说,这意味着:
- 如果你只会"快速做东西",你的价值会降低——因为AI也能
- 如果你能在"快速做"之后"深度思考、系统探索",你的价值会提升——因为这是AI做不到的

**技术工具的进步,最终拼的是人的能力。** Claude Code降低了写的门槛,Figma降低了改的门槛,但:
- 决策哪个方案更好,需要你的产品洞察
- 设计更优雅的交互,需要设计师的专业能力
- 写出可维护的代码,需要工程师的架构思维

**工具在进化,人的价值也在进化。**
